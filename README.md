# Network Traffic Analysis Report

**Author:** Jacob Pinkerton | **Date:** May 2026 | **Tools:** Wireshark, Windows 11

---

## Executive Summary

A passive network traffic capture was conducted on a home network environment using Wireshark. The capture ran for approximately 10 minutes while the host machine was largely idle, connected to a Discord voice call. A total of 372,673 packets were captured. Analysis focused on four protocol areas: DNS, HTTP, DHCP, and TLS. The capture revealed normal background traffic patterns consistent with active applications running on idle devices, however unencrypted HTTP traffic was identified as a minor point of interest. No malicious or anomalous traffic was detected during the capture window. TLS version distribution was predominantly TLSv1.2 with some TLSv1.3 present, which is worth noting from a security hardening perspective.

---

## Scope & Methodology

- **Capture interface:** Primary network adapter (home network)
- **Tool:** Wireshark
- **Capture duration:** Approximately 10 minutes
- **Host activity during capture:** Largely idle, active Discord voice call
- **Total packets captured:** 372,673
- **Filters applied:** `dns`, `http`, `dhcp`, `tls`

*IP addresses, MAC addresses, and device identifiers have been anonymised in this report for operational security purposes.*

## Detailed Analysis

### DNS Traffic

The DNS filter returned 1,402 packets, representing 0.4% of the total capture. Queries were observed to several external destinations consistent with applications running passively in the background on the host machine. Identified destinations included:

- **Warframe (api.warframe.market)** — game client polling in the background
- **Spotify (spclient.wg.spotify.com)** — passive keep-alive and service queries
- **Anthropic (assets-proxy.anthropic.com, a-api.anthropic.com)** — background service calls consistent with an active browser session
- **Steam (p2p-syd1.discovery.steamserver.net)** — peer discovery queries from the Steam client
- **Microsoft (array511.prod.do.dsp.mp.microsoft.com)** — Windows background service traffic

Both A (IPv4) and AAAA (IPv6) record queries were observed, indicating the network supports dual-stack operation. All queries resolved normally with expected responses from Google's public DNS resolver (8.8.8.8).

**Observation:** The volume and variety of DNS queries generated by an idle machine highlights how much background communication modern applications perform without any direct user interaction. From a security monitoring perspective, unexpected domains appearing in this list would be an early indicator of potentially unwanted software or compromised endpoints.

---

### HTTP Traffic

The HTTP filter returned 90 packets, representing a small but notable portion of traffic. Unencrypted HTTP was observed in several areas including certificate revocation list checks, internal device communication between the host and router, and background application calls.

Notable HTTP activity included:

- **DigiCert CRL check (GET /DigiCertTrustedRoot64.crl)** — standard certificate validation, expected behaviour
- **Spotify app calls (GET /apps/com.spotify.Spotify.TVv2)** — application background polling over HTTP rather than HTTPS
- **Google Analytics pings** — background telemetry from browser activity
- **Router communication (POST /ctl/IPConn)** — internal HTTP traffic between the host and the residential gateway on port 80

**Finding:** The presence of HTTP traffic, while not unusual in a home environment, is worth flagging. Any communication occurring over HTTP rather than HTTPS is unencrypted and readable in plaintext by anyone on the same network. In a business environment, this would be flagged for remediation, particularly application traffic that should be using HTTPS by default.

---

### DHCP Traffic

The DHCP filter returned zero packets during the capture window. This is expected behaviour rather than an absence of network activity. DHCP leases on a home network typically last several hours, meaning devices already connected at the start of the capture would not generate renewal traffic within a 10 minute window.

**Observation:** To capture DHCP traffic a longer capture window would be needed, or a device would need to join the network fresh during the capture period. The absence of results here is normal and does not indicate a misconfiguration.

---

### TLS Traffic

The TLS filter returned 48,306 packets, representing 13.0% of total captured traffic and by far the largest filtered dataset. This is expected given that the majority of modern internet traffic is encrypted.

Key observations:

- **TLSv1.2** was the dominant protocol version observed across the majority of connections
- **TLSv1.3** was observed on some connections, visible via Server Hello handshakes in the capture
- An **Ignored Unknown Record** entry was noted on one TLSv1.2 connection, which can indicate a non-standard TLS implementation or a minor handshake anomaly, though this alone is not a cause for concern
- External destinations observed included IP ranges consistent with Microsoft, Spotify, and other services identified in the DNS analysis, confirming the traffic sources

**Observation:** The predominance of TLSv1.2 rather than TLSv1.3 is worth noting from a hardening perspective. TLSv1.3 offers improved security and performance over TLSv1.2, and while TLSv1.2 is not considered insecure, a fully hardened environment would aim to migrate to TLSv1.3 across all connections over time. TLSv1.0 and TLSv1.1 were not observed, which is a positive indicator.

---

## Key Observations Summary

- Background application traffic is continuous even on an idle machine. Several applications including Steam, Spotify, and Warframe were generating DNS queries and network connections without any direct user interaction.
- Unencrypted HTTP traffic is still present on the network, primarily from internal router communication and some application background calls. While low volume, this represents a minor attack surface on any shared network.
- DNS queries provide a clear picture of what applications are active on a host even without decrypting any traffic. This is a core technique used in network security monitoring and threat hunting.
- TLS adoption is strong across the capture, with no deprecated TLS versions observed. Migration toward TLSv1.3 would further improve the security posture.
- DHCP produced no results in this window, which is normal for a short passive capture on an established network.

---

## Recommendations

- Where possible, ensure applications are configured to use HTTPS rather than HTTP for all external communication
- Monitor DNS query logs regularly as a low-overhead method of identifying unexpected application behaviour or potentially unwanted software
- Consider enforcing TLSv1.3 as the minimum TLS version on devices and services where configuration allows
- For future captures, extend the capture window and introduce a new device to the network to capture the full DHCP handshake process

---

## Conclusion

This passive capture provided a clear picture of normal background traffic behaviour on a home network. No malicious or anomalous activity was identified during the capture window. The most notable finding was the presence of unencrypted HTTP traffic, which while low volume, represents an area for improvement. The DNS analysis demonstrated how much network activity modern applications generate passively, reinforcing the value of DNS monitoring as a lightweight detection technique. A follow-up capture with a longer window and broader scope across multiple devices would provide a more comprehensive view of the network's overall traffic profile.
